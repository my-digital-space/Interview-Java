PTD. Shallow Copy vs Deep Copy

 

A.

 

In Java, copying an object can be done in two primary ways: shallow copy and deep copy. These concepts are fundamental when working with object-oriented programming, as they determine how data is duplicated and whether the original and copied objects remain independent.

 

Shallow Copy

A shallow copy creates a new object that is a duplicate of the original object, but it only copies the references to any objects contained within the original. This means that the new object points to the same nested objects as the original. If those nested objects are mutable (i.e., their state can be changed), modifications to them in the copy will also affect the original, and vice versa.

 

Key Characteristics:

Copies the top-level fields of the object.

References to nested objects are shared between the original and the copy.

Changes to the internal objects affect both the original and the copy.

 

 

Deep Copy

A deep copy, on the other hand, creates a completely independent duplicate of the original object. It recursively copies all objects referenced by the original, ensuring that the copy has its own separate instances of all nested objects. As a result, changes to the copy do not affect the original, and vice versa.

 

Key Characteristics:

Creates a new object and duplicates all nested objects recursively.

The copy is fully independent of the original.

Changes to the copy‚Äôs internal objects do not impact the original.

How to Create an Immutable Class in Java? And when to use it?

 

A.

 

An immutable class in Java is a class whose instances cannot be modified after creation.

 

Steps to make a class as an immutable class:

Step 1: Declare the Class as ‚Äúfinal‚Äù

Step 2: Make All Fields ‚Äúprivate‚Äù and ‚Äúfinal‚Äù

Step 3: Initialize All Fields via a public Constructor

Step 4: Do Not Provide Setter Methods

Step 5: Perform Defensive Copies for Mutable Objects

 

--------------------------------------------------------------------------------------------------------

Step 1: Declare the Class as ‚Äúfinal‚Äù

Marking the class as final prevents it from being subclassed, which helps in preserving immutability.

 

--------------------------------------------------------------------------------------------------------

Step 2: Make All Fields ‚Äúprivate‚Äù and ‚Äúfinal‚Äù

All instance variables should be declared as private and final so that their values can be assigned only once.

Provide a public constructor that initializes all fields. Since the fields are final, they can only be assigned once.

 

--------------------------------------------------------------------------------------------------------

Avoid any methods that modify fields or return a reference to mutable objects directly.

 

--------------------------------------------------------------------------------------------------------

Step 5: Perform Defensive Copies for Mutable Objects

If your class holds mutable objects (like Date, collections, etc.), create defensive copies in the constructor and in the getter methods.

Since String is immutable in Java, whenever we do String manipulation like concatenation, substring, etc. it generates a new String and discards the older String for garbage collection. StringBuffer was the only choice for String manipulation until Java 1.4. But, it has one disadvantage that all of its public methods are synchronized. StringBuffer has some extra methods such as substring, length, capacity, trimToSize, etc. However, these are not required since you have all these present in String too. That‚Äôs why these methods were never implemented in the StringBuilder class. StringBuffer was introduced in Java 1.0 whereas StringBuilder class was introduced in Java 1.5 after looking at shortcomings of StringBuffer. If you are in a single-threaded environment or don‚Äôt care about thread safety, you should use StringBuilder. Otherwise, use StringBuffer for thread-safe operations.

String a = new String("Hello"); vs String a = "Hello";

 

A.

 

When you write:

1. String a = new String("Hello");

...a new String object is created in memory, regardless of whether the string literal "Hello" already exists in the string pool.

 

In contrast, when you write:

1. String a = "Hello";

...the string literal "Hello" is placed in the string pool. If an identical string already exists there, the reference is reused rather than creating a new object.

 

Key Differences

Memory Allocation:

new String("Hello"): Always creates a new object on the heap.

"Hello" Literal: Uses the interned string from the string pool, reducing memory footprint.

Reference Equality:

Using == to compare a string created with new String("Hello") and a literal "Hello" will return false since they reference different objects, even though their content is identical.

Performance:

Reusing literals via the string pool can be more efficient, as it avoids creating duplicate objects.

 

Example

1. String s1 = new String("Hello");

2. String s2 = "Hello";

3. ¬†

4. System.out.println(s1 == s2); // Outputs: false (different references)

5. System.out.println(s1.equals(s2)); // Outputs: true (same content)

6. ¬†

7. // Using intern() to force pooling:

8. String s3 = s1.intern();

9. System.out.println(s2 == s3); // Outputs: true (both point to the pooled literal)

== vs .equals() while comparing two objects?

 

A.

 

In Java, the == operator and the .equals() method are used for comparisons, but they serve different purposes:

== Operator:

Reference Comparison: For objects, == checks if both references point to the exact same memory location.

Primitive Comparison: When used with primitive types (like int, char, etc.), it compares the actual values.

.equals() Method:

Logical/Content Comparison: The .equals() method is intended to compare the content or state of two objects.

Overridable: By default (as defined in the Object class), .equals() behaves like == (i.e., it checks reference equality). However, many classes (like String, Integer, etc.) override this method to provide a logical equality check.

Example

1. String a = new String("Hello");

2. String b = new String("Hello");

3. ¬†

4. // Using '==': compares memory addresses, so this is false

5. System.out.println(a == b); // Output: false

6. ¬†

7. // Using '.equals()': compares the content of the strings, so this is true

8. System.out.println(a.equals(b)); // Output: true

9. ¬†

 

Key Takeaways

Use == when you need to check if two references point to the same object instance.

Use .equals() when you want to check if two objects are logically equivalent based on their state or content.

Always consider overriding .equals() (and hashCode()) in your custom classes if you want to define what it means for two instances to be "equal."

 

This distinction is critical for ensuring that your Java programs compare objects correctly, especially when dealing with collections or custom objects.

Q. Error vs Exception

 

A.

 

Errors

Definition: Errors indicate serious problems that the application typically cannot recover from. These are generally external to the application‚Äôs logic and are thrown by the Java Virtual Machine (JVM).

Hierarchy: Errors are subclasses of the java.lang.Error class.

Examples:

OutOfMemoryError: Occurs when the JVM runs out of memory.

StackOverflowError: Occurs when the stack memory allocated to a thread is exceeded, often because of deep or infinite recursion.

Handling: Applications are generally not expected to catch errors because they usually indicate unrecoverable conditions. Instead, they may signal a problem with the runtime environment.

 

Exceptions

Definition: Exceptions represent conditions that a program might want to catch and handle gracefully. They are usually problems that can occur during normal program execution, such as invalid input or failure to open a file.

Hierarchy: Exceptions are subclasses of the java.lang.Exception class.

Types:

Checked Exceptions:

Must be either caught or declared in the method's throws clause.

Example: FileNotFoundException, IOException

Unchecked Exceptions (Runtime Exceptions):

These are not required to be caught or declared. They generally indicate programming bugs, such as logic errors or improper use of an API.

Example: NullPointerException, ArrayIndexOutOfBoundsException

Handling: Exceptions can be caught using try-catch blocks, allowing the program to recover or take alternative actions when an error condition occurs.

Q. Why should we avoid catching errors?

 

A.

 

Catching errors in Java is generally discouraged because they represent serious problems that your application is not designed to handle.

 

1. Indication of Serious Problems

System-Level Issues: Errors such as OutOfMemoryError or StackOverflowError are thrown by the JVM to indicate that the system is in a critical state. These issues typically arise from severe resource limitations or programming mistakes (like infinite recursion) that compromise the entire application's stability.

Beyond Application Control: Errors are not meant to be recoverable by application code. Their occurrence usually means that the runtime environment has encountered a problem that falls outside the bounds of normal execution. Attempting to catch such errors might lead to unpredictable behavior because the underlying system may not be in a state to safely continue.

 

2. Potential for Inconsistent State

Unreliable Program State: When an error occurs, it often means that the JVM's internal state or system resources have been compromised. For example, if an OutOfMemoryError is thrown, it might be impossible to safely allocate further objects or perform operations that depend on memory. Catching the error might allow the program to continue running, but in a corrupted or unpredictable state.

Difficulty in Recovery: Even if you catch an error, there is rarely a safe or meaningful way to recover from it. Unlike exceptions, which might be anticipated and handled (for instance, by retrying a network operation), errors usually signal that something is fundamentally wrong with the environment, and recovery might require restarting the application or even the system.

 

3. Masking Underlying Issues

Hiding Critical Failures: By catching errors, you risk obscuring serious problems that should be visible to developers or operations teams. If errors are caught and not allowed to propagate, debugging becomes more difficult because the application might continue running in an inconsistent state while the root cause remains unaddressed.

Unintentional Continuation: Handling errors might inadvertently lead the program to continue executing when it should instead terminate. This can lead to further complications, such as data corruption, security vulnerabilities, or more complex failures that are harder to diagnose and fix.

 

4. Best Practices and JVM Guidelines

Design Philosophy: The Java language differentiates between errors and exceptions to indicate that errors are not meant to be recovered from by typical application logic. The design philosophy behind this is to ensure that critical issues are handled by the JVM or by system-level mechanisms (like logging or graceful shutdown) rather than by the application code itself.

Recommended Approach: Instead of catching errors, it‚Äôs better to let them propagate and cause the application to fail fast. This allows your error-handling infrastructure (such as a top-level error handler) to log the problem, perform necessary cleanup, and, if possible, restart the application. This is a more robust strategy than attempting to recover from a state that may be unstable.

 

 

Conclusion

Catching errors in Java is discouraged because:

Errors signal critical issues that are beyond the scope of typical application logic.

Recovering from errors is risky due to the potential for inconsistent or corrupted program state.

Catching errors may mask real problems, making them harder to diagnose and resolve.

Best practices dictate that errors should be allowed to propagate, letting the system or dedicated error-handling mechanisms manage them.

Q. Checked vs Unchecked exception

 

A.

 

Java exceptions are broadly classified into two types: checked exceptions and unchecked exceptions. Let‚Äôs break down the differences step by step and illustrate with both built-in and custom examples.

3. Key Differences Recap

throw:

Action: Actively throws an exception.

Usage: Inside the method/block where the exception condition is met.

Example: throw new IllegalArgumentException("Invalid input");

throws:

Declaration: Indicates that a method might throw one or more exceptions.

Usage: In the method signature.

Example: public void readFile(String fileName) throws FileNotFoundException

 

4. When to Use Each

Use throw: When you want to explicitly trigger an exception in your code. For example, validating method parameters and throwing an exception when the input is not acceptable.

Use throws: When your method calls other methods that might throw exceptions (especially checked exceptions) and you want to pass the responsibility of handling those exceptions to the caller.

This behavior is exactly what the Stream API is designed to do: it doesn‚Äôt execute one whole stage (all the ‚Äúbefore‚Äù peeks) and then another whole stage (all the ‚Äúafter‚Äù peeks). Instead, it processes each element end-to-end, one at a time, passing it through all intermediate stages before pulling the next element.

 

Concretely, for your sequential stream:

The terminal operation (the collect) requests the next element.

That element is sent down the pipeline:

peek(before) logs ‚ÄúBefore filter: n‚Äù

filter tests n % 2 == 0

peek(after) logs ‚ÄúAfter filter: n‚Äù only if the filter passed

Only once that element has traversed every stage does the stream pull the next element.

So you see:

Before filter: 1 (1 fails the filter; no ‚ÄúAfter filter‚Äù for it)

Before filter: 2 ‚Üí After filter: 2

Before filter: 3 (3 fails; no after)

Before filter: 4 ‚Üí After filter: 4

‚Ä¶and so on.

If you wanted to group all the ‚Äúbefore‚Äù logs together, you‚Äôd have to break the pipeline into two separate passes‚Äîfirst peeking and collecting or buffering all elements, then running the filter + after-peek on that collection. But that defeats the point of the one-pass, lazy, fused nature of streams.

QTD. What is CompletableFuture? And why is it required?

Q. What are the limitations of the traditional Future?

 

A. TO DO

CompletableFuture is a powerful feature introduced in Java 8 as part of the¬†java.util.concurrent¬†package. It enhances the capabilities of the traditional¬†Future¬†interface, allowing for more flexible and efficient asynchronous programming.

 

Why is CompletableFuture Required? | Limitations of the Traditional Future

While the original¬†Future¬†interface provided a way to handle asynchronous tasks, it had several limitations:

Manual Completion: Futures cannot be manually completed once created. If a task needs to be completed based on an external condition (e.g., an API call failure), it cannot be done with a standard Future.

Chaining Limitations: With Futures, you cannot chain multiple tasks together. If one task needs to feed into another, you must manage this manually, which complicates code readability and maintenance.

Combining Futures: Futures do not support combining multiple asynchronous tasks easily. For instance, if you want to run several tasks in parallel and then process their results together, this is cumbersome with Futures.

Poor Exception Handling: The Future API lacks built-in exception handling capabilities. You must catch exceptions during the retrieval of results, which can lead to unmanageable code in complex scenarios.

 

 

 

 

 

1. Manual Completion

Future Limitation: A standard Future‚Äôs result is determined solely by the task execution. You cannot ‚Äúforce‚Äù its completion based on external conditions.

CompletableFuture Advantage: You can manually complete a CompletableFuture using methods like complete() or completeExceptionally().

 

 

 

 

2. Chaining Asynchronous Operations

Future Limitation: To chain tasks with Future, you must block (using get()) and then start the next task manually. This breaks the asynchronous flow and clutters the code.

CompletableFuture Advantage: Chaining is built-in with methods like thenApply() and thenCompose(), enabling a smooth, non-blocking pipeline.

 

 

 

3. Combining Multiple Futures

Future Limitation: With Future, if you run tasks in parallel, you must wait (block) for each to finish and then combine their results manually.

CompletableFuture Advantage: Methods like thenCombine(), allOf(), or anyOf() allow you to merge multiple asynchronous results elegantly.

 

 

 

4. Exception Handling in Asynchronous Tasks

Future Limitation: When a task throws an exception, you must catch it after calling get(), which can lead to messy error handling.

CompletableFuture Advantage: Built-in methods like exceptionally() and handle() allow you to manage exceptions in a fluent, non-blocking manner.

TD. Explain the JVM Architecture

 

A.

 

Overview:

The Java Virtual Machine (JVM) architecture includes several key components for running Java programs.

The main components are Class Loader, Runtime Memory Areas, Execution Engine, Garbage Collector, and Java Native Interface.

These components work together to load, execute, and manage Java applications efficiently.

 

JVM‚Äôs 5 main Components:

1. Class Loader

2. Runtime Memory Areas

3. Execution Engine

4. Garbage Collector

5. Java Native Interface (JNI)

 

1. Class Loader

This part loads Java classes into the JVM. It includes:

Bootstrap Class Loader: Loads core Java classes, like those in the Java Runtime Environment.

Extension/Platform Class Loader: Loads additional libraries, often from specific directories.

Application Class Loader: Loads the classes you write for your application.

Note: In Java 9 and later, the Extension Class Loader is called the Platform Class Loader, reflecting changes in how classes are managed.

 

2. Runtime Memory Areas

These are memory spaces used while the program runs:

Heap Memory: Stores objects created during execution.

Stack Area: Manages method calls and local variables for each thread.

Method Area: Holds method code and static variables, shared across threads.

Program Counter (PC) Register: Tracks the current instruction being executed for each thread.

Native Method Stack: Handles native methods, which are written in languages like C or C++.

 

3. Execution Engine

This executes the Java bytecode:

Interpreter: Reads and runs bytecode line by line, which can be slower for repeated tasks.

Just-in-Time (JIT) Compiler: Compiles frequently used code into machine code for faster performance.

 

4. Garbage Collector

This automatically frees up memory by removing unused objects:

Serial Garbage Collector: Uses one thread, best for small, single-threaded applications.

Parallel Garbage Collector: Uses multiple threads, suitable for multi-core systems.

Other types include Concurrent Mark-Sweep, G1, Z, and Shenandoah, each designed for different needs like low pause times or large heaps.

 

5. Java Native Interface (JNI)

This allows Java to interact with code written in other languages, mainly C and C++, and potentially others with C APIs, enabling access to system-specific features.

What does the statement ‚Äúmemory is managed in Java‚Äù mean?

Memory is the key resource an application requires to run effectively and like any resource, it is scarce. As such, its allocation and deallocation to and from applications or different parts of an application require a lot of care and consideration.

However, in Java, a developer does not need to explicitly allocate and deallocate memory ‚Äì the JVM and more specifically the Garbage Collector ‚Äì has the duty of handling memory allocation so that the developer does not have to.

This is contrary to what happens in languages like C where a programmer has direct access to memory and literally references memory cells in his code, creating a lot of room for memory leaks.

 

 

What is garbage collection and what are its advantages?

 

References:

https://www.baeldung.com/java-memory-management-interview-questions

 

 

Garbage collection is the process of looking at heap memory, identifying which objects are in use and which are not, and deleting the unused objects.

An in-use object, or a referenced object, means that some part of your program still maintains a pointer to that object. An unused object, or unreferenced object, is no longer referenced by any part of your program. So, the memory used by an unreferenced object can be reclaimed.

 

The biggest advantage of garbage collection is that it removes the burden of manual memory allocation/deallocation from us so that we can focus on solving the problem at hand.

 

 

Are there any disadvantages of garbage collection?

 

References:

https://www.baeldung.com/java-memory-management-interview-questions

 

 

Yes. Whenever the garbage collector runs, it has an effect on the application‚Äôs performance. This is because all other threads in the application must be stopped to allow the garbage collector thread to effectively do its work.

 

Depending on the requirements of the application, this can be a real problem that is unacceptable by the client. However, this problem can be greatly reduced or even eliminated through skilful optimization and garbage collector tuning and using different GC algorithms.

 

 

 

What is the meaning of the term ‚ÄúStop-The-World‚Äù?

 

References:

https://www.baeldung.com/java-memory-management-interview-questions

 

 

When the garbage collector thread is running, other threads are stopped, meaning the application is stopped momentarily. This is analogous to house cleaning or fumigation where occupants are denied access until the process is complete.

 

Depending on the needs of an application, ‚Äústop the world‚Äù garbage collection can cause an unacceptable freeze. This is why it is important to do garbage collector tuning and JVM optimization so that the freeze encountered is at least acceptable.

 

 

What are Stack and Heap? What is stored in each of these memory structures, and how are they interrelated?

 

References:

https://www.baeldung.com/java-memory-management-interview-questions

 

 

The stack is a part of memory that contains information about nested method calls down to the current position in the program. It also contains all local variables and references to objects on the heap defined in currently executing methods.

This structure allows the runtime to return from the method knowing the address whence it was called, and also clear all local variables after exiting the method. Every thread has its own stack.

 

The heap is a large bulk of memory intended for allocation of objects. When you create an object with the¬†new¬†keyword, it gets allocated on the heap. However, the reference to this object lives on the stack.

 

 

What Is Generational Garbage Collection and what makes it a popular Garbage Collection approach?

 

References:

https://www.baeldung.com/java-memory-management-interview-questions

 

 

Generational garbage collection can be loosely defined as the strategy used by the garbage collector where the heap is divided into a number of sections called generations, each of which will hold objects according to their ‚Äúage‚Äù on the heap.

Whenever the garbage collector is running, the first step in the process is called marking. This is where the garbage collector identifies which pieces of memory are in use and which are not. This can be a very time-consuming process if all objects in a system must be scanned.

As more and more objects are allocated, the list of objects grows and grows leading to longer and longer garbage collection time. However, empirical analysis of applications has shown that most objects are short-lived.

With generational garbage collection, objects are grouped according to their ‚Äúage‚Äù in terms of how many garbage collection cycles they have survived. This way, the bulk of the work spread across various minor and major collection cycles.

Today, almost all garbage collectors are generational. This strategy is so popular because, over time, it has proven to be the optimal solution.

 

 

Describe in detail how Generational Garbage Collection works

 

References:

https://www.baeldung.com/java-memory-management-interview-questions

 

 

To properly understand how generational garbage collection works, it is important to first¬†remember how Java heap is structured¬†to facilitate generational garbage collection.

The heap is divided up into smaller spaces or generations. These spaces are Young Generation, Old or Tenured Generation, and Permanent Generation.

The¬†young generation hosts most of the newly created objects. An empirical study of most applications shows that majority of objects are quickly short lived and therefore, soon become eligible for collection. Therefore, new objects start their journey here and are only ‚Äúpromoted‚Äù to the old generation space after they have attained a certain ‚Äúage‚Äù.

The term¬†‚Äúage‚Äù¬†in generational garbage collection¬†refers to the number of collection cycles the object has survived.

The young generation space is further divided into three spaces: an Eden space and two survivor spaces such as Survivor 1 (s1) and Survivor 2 (s2).

The¬†old generation hosts objects that¬†have lived in memory longer than a certain ‚Äúage‚Äù. The objects that survived garbage collection from the young generation are promoted to this space. It is generally larger than the young generation. As it is bigger in size, the garbage collection is more expensive and occurs less frequently than in the young generation.

The¬†permanent generation¬†or more commonly called,¬†PermGen,¬†contains metadata required by the JVM¬†to describe the classes and methods used in the application. It is populated by the JVM at runtime based on classes in use by the application. In addition, platform library classes and methods may be stored here.

First,¬†any new objects are allocated to the Eden space. Both survivor spaces start out empty. When the Eden space fills up, a minor garbage collection is triggered. Referenced objects are moved to the first survivor space. Unreferenced objects are deleted.

During the next minor GC, the same thing happens to the Eden space. Unreferenced objects are deleted and referenced objects are moved to a survivor space. However, in this case, they are moved to the second survivor space (S2).

In addition, objects from the last minor GC in the first survivor space (S1) have their age incremented and are moved to S2. Once all surviving objects have been moved to S2, both S1 and Eden space are cleared. At this point, S2 contains objects with different ages.

At the next minor GC, the same process is repeated. However, this time the survivor spaces switch. Referenced objects are moved to S1 from both Eden and S2. Surviving objects are aged. Eden and S2 are cleared.

After every minor garbage collection cycle, the age of each object is checked. Those that have reached a certain arbitrary age, for example, 8, are promoted from the young generation to the old or tenured generation. For all subsequent minor GC cycles, objects will continue to be promoted to the old generation space.

This pretty much exhausts the process of garbage collection in the young generation. Eventually, a major garbage collection will be performed on the old generation which cleans up and compacts that space. For each major GC, there are several minor GCs.
